# Tilo Framework v0.1 Codex Development Specification

## 1. Product Vision

Tilo Framework is an open-source framework for building memory-native, self-improving AI agents that can execute real work and deliver SaaS-like interactive artifacts.

The framework should not be designed as a chatbot wrapper. It should be designed as an AI-native SaaS runtime:

```text
Conversation -> Agent Task -> Memory Recall -> Tool Execution -> Artifact Delivery -> Human Confirmation -> Memory Update
```

## 2. Core Principles

### Conversation as Command

User messages are commands or goals, not just chat turns. The system should transform user intent into structured tasks.

### Artifact as Product

Agent outputs should become structured artifacts, not only Markdown messages. Artifacts should be editable, versioned, and renderable in the Web Console.

### Human as Decision Maker

Humans should only be asked to confirm, approve, edit, or reject important actions. The system should avoid forcing users to operate every workflow step manually.

### Memory-first Agents

Long-term memory is a first-class system capability. Agents should remember users, projects, decisions, preferences, and reusable procedures.

### Self-improving Loop

After task completion, the system should extract useful memory candidates and optionally propose skill improvements.

## 3. MVP Goal

Build a complete but lightweight v0.1 loop:

1. User sends a message in the Web Console.
2. Backend creates a Task and a Run.
3. Runtime recalls related Memory.
4. Runtime builds a simple execution plan.
5. Runtime invokes tools or mocked tools.
6. Runtime generates an Artifact.
7. Runtime creates Confirmation items when needed.
8. User approves, rejects, or edits confirmations.
9. Run completes.
10. System extracts memory candidates.
11. User confirms memory candidates.
12. Confirmed memories are stored and can be recalled in future tasks.

## 4. Recommended Tech Stack

## 7. Backend API Requirements

Implement REST APIs for v0.1.

- `GET /api/workspaces`
- `POST /api/workspaces`
- `GET /api/workspaces/{id}`
- `PATCH /api/workspaces/{id}`

### Projects

- `GET /api/projects?workspace_id=`
- `POST /api/projects`
- `GET /api/projects/{id}`
- `PATCH /api/projects/{id}`

### Agents

- `GET /api/agents?workspace_id=`
- `POST /api/agents`
- `GET /api/agents/{id}`
- `PATCH /api/agents/{id}`

### Tasks and Runs

- `POST /api/tasks`
- `GET /api/tasks?workspace_id=&project_id=`
- `GET /api/tasks/{id}`
- `POST /api/tasks/{id}/runs`
- `GET /api/runs/{id}`
- `GET /api/runs/{id}/trace`

### Chat / Message Entry

- `POST /api/messages`

This endpoint should accept:

```json
{
  "workspace_id": "...",
  "project_id": "...",
  "agent_id": "...",
  "content": "Help me review this contract",
  "attachments": []
}
```

### Artifacts

- `PATCH /api/artifacts/{id}`

### Confirmations / Inbox

- `POST /api/confirmations/{id}/reject`

### Skills

- `GET /api/skills?workspace_id=`

- `GET /api/skills/{id}`

### Tools

- `GET /api/tools?workspace_id=`

- `GET /api/tools/{id}`

## 8. Agent Runtime Requirements

Implement a lightweight runtime with these services:

### Prompt Builder

Builds model input from:
- agent system prompt
- user task
- recalled memories
- available skills
- available tools
- artifact output requirements

### Planner

For v0.1, a simple LLM-based or rule-based planner is enough.

Output shape:

```json
{
  "goal": "...",
  "steps": [
    {"type": "recall_memory", "description": "..."},
    {"type": "invoke_tool", "description": "..."},
    {"type": "generate_artifact", "description": "..."},
    {"type": "ask_confirmation", "description": "..."}
  ]
}
```

### Executor

Executes plan steps and records TraceStep records.

### Tool Router

Routes tool invocations to registered tool implementations.

### Artifact Generator

Generates artifact schema JSON. Do not hardcode everything into Markdown.

### Memory Extractor

After run completion, extracts memory candidates from:
- user input
- task result
- user confirmations
- final artifact

Memory candidates should start as `is_confirmed=false`.

## 9. Artifact Schema v0.1

Artifacts must use a common JSON shape:

```json
{
  "artifact_type": "document",
  "title": "Example Artifact",
  "blocks": [
    {
      "id": "block_1",
      "type": "markdown",
      "data": {
        "content": "..."
      }
    }
  ]
}
```

Supported block types:
- markdown
- table
- card
- metric
- list
- kanban
- timeline
- risk_item
- confirmation_action

### Document Artifact

Use markdown blocks and summary cards.

### Table Artifact

Use column definitions and row data.

### Dashboard Artifact

Use metric cards, insight cards, and lists.

### Kanban Artifact

Use columns and cards.

### Contract Review Artifact

Use:
- document summary
- clause risk list
- risk level
- suggested revision
- confirmation action

## 11. Initial Demo Requirements

### Demo 1: Contract Review Agent

Input:
- User uploads or pastes contract text.

Expected behavior:
- Agent identifies contract type.
- Agent extracts key clauses.
- Agent flags risks.
- Agent generates suggested revisions.
- Agent creates a contract_review artifact.
- Agent creates confirmation items for high-risk suggested actions.
- Agent extracts memory candidates about user risk preferences.

### Demo 2: Sales Follow-up Agent

Input:
- User asks which customers should be followed up.

Expected behavior:
- Agent uses mock CRM data.
- Agent scores customer opportunities.
- Agent generates follow-up recommendations.
- Agent creates confirmation items for sending follow-up messages.
- Agent generates a dashboard artifact.

### Demo 3: Competitive Analysis Agent

Input:
- User asks for competitor research.

Expected behavior:
- Agent uses mock search data or user-provided text.
- Agent generates comparison table.
- Agent summarizes market opportunities.
- Agent creates a document + table artifact.

## 12. Security Requirements

v0.1 must include basic safety design:

- Tool permission levels: low, medium, high.
- High-risk tools must create Confirmation before execution.
- All tool calls must be logged as TraceStep.
- Secrets must never be shown in trace output.
- Do not expose hidden model reasoning.
- Prompt injection defense should be documented even if basic in v0.1.

## 14. Acceptance Criteria

Codex should consider v0.1 complete when:

1. The repo has backend and frontend skeletons.
2. Docker Compose can start backend, frontend, Postgres, and Redis.
3. The Web Console can send a user message.
4. A Task and Run are created.
5. Runtime creates trace steps.
6. Runtime recalls existing memories.
7. Runtime generates at least one Artifact.
8. Artifact renders in the frontend.
9. Confirmation items appear in Inbox.
10. User can approve or reject a confirmation.
11. Memory candidates are created after task completion.
12. User can confirm memory candidates.
13. Confirmed memories can be recalled in a future task.
14. At least one demo flow works end to end.

## 15. Development Order

Recommended order:

1. Create backend FastAPI skeleton.
2. Create database models and migrations.
3. Create frontend Next.js skeleton.
4. Implement workspace/project/agent CRUD.
5. Implement task/run/message APIs.
6. Implement trace system.
7. Implement artifact schema and renderer.
8. Implement confirmation inbox.
9. Implement memory storage and recall.
10. Implement memory extraction candidates.
11. Implement skill placeholder system.
12. Implement tool registry and mock tools.
13. Implement contract review demo.
14. Add Docker Compose and documentation.

## 16. Non-goals for v0.1

Do not overbuild:

- Full multi-agent collaboration.
- Enterprise RBAC.
- Production-grade distributed task scheduling.
- Full MCP marketplace.
- Real browser automation.
- Real email sending or external destructive actions.
- Automatic self-modifying code.

v0.1 should prioritize a complete product loop over local perfection.
