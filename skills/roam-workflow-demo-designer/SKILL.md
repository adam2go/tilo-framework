# ROAM Workflow Demo Designer Skill

Use this skill when designing or implementing Tilo's public-facing demo, ROAM Workspace, AI-native SaaS Workflow Surface, interaction components, onboarding, or README-ready screenshots.

This skill is written for Codex, Claude Code, and human contributors.

---

## 1. Role

You are a senior AI-native SaaS product designer, frontend architect, and open-source demo designer.

Your goal is to make Tilo's demo instantly understandable, visually polished, and conceptually differentiated.

Do not merely make the existing UI prettier.

Your job is to prove this product idea:

```text
Agents can generate interactive SaaS workflows, observe human decisions, act safely, and memorize confirmed learning.
```

---

## 2. Core Concept

Tilo's demo is built around the ROAM Loop:

```text
Render -> Observe -> Act -> Memorize
```

The main UI should make this loop visible.

Every demo interaction should answer:

1. What did the agent render?
2. What did the user observe or decide?
3. What action happens next?
4. What can Tilo memorize for the future?

---

## 3. Product Form

The primary product form is:

```text
Conversation-first ROAM Workspace
```

Not:

- a plain chatbot
- a full infinite canvas
- a generic admin dashboard
- a static report viewer
- a debug console

The workspace should combine:

```text
Left: Conversation / Goal Steering
Center: Generated SaaS Workflow Surface
Right: Context / Trace / Memory / Inbox / Skills / Observations
```

---

## 4. Visual Quality Target

Aim for:

```text
Linear clarity + Vercel polish + Notion structure + Raycast speed
```

This means:

- clean layout
- strong hierarchy
- calm professional palette
- crisp typography
- high information density without clutter
- obvious primary actions
- premium open-source feel

Avoid:

- admin dashboard grayness
- raw JSON blocks
- toy gradients
- large unstyled text areas
- generic Tailwind starter look
- cramped spacing
- inconsistent buttons

---

## 5. Layout Requirements

### Topbar

Should show:

- Tilo logo/name
- ROAM Loop label
- Workspace/project context
- demo status
- GitHub or Docs link if appropriate

### Left Panel: Conversation / Goal Steering

Should show:

- short headline
- goal input
- demo prompt cards
- run button
- run progress
- suggested next action

This area should feel like a mission control panel, not a chat toy.

### Center Panel: Generated SaaS Workflow Surface

This is the hero.

It must show a sequential business workflow:

```text
Contract Input -> Risk Review -> Human Approval -> Revision Draft -> Memory Update -> Next Action
```

Use workflow nodes, cards, badges, and interaction components.

### Right Panel: Context / Control

Should show tabs:

- Trace
- Memory
- Inbox
- Skills
- Observations

Use badges for pending confirmations and memory candidates.

---

## 6. Contract Review Demo Requirements

The main demo is Contract Review.

It should look like an AI-native legal SaaS workflow generated by an agent.

Required components:

### RiskSummary

Show:

- high risk count
- medium risk count
- low risk count
- short assessment
- review status

### RiskReviewPanel

Show several realistic clause risks:

- Payment terms
- Liability limitation
- Termination rights
- Confidentiality
- IP ownership

Each risk item should include:

- risk level badge
- issue summary
- suggested revision
- approve/edit/reject actions

### ApprovalCard

Show a pending human decision:

```text
Generate a conservative revision draft based on these risks?
```

Actions:

- Approve
- Edit before approve
- Reject

### EditableDocumentPreview

Show a polished revision preview.

It does not need to be a full editor yet, but should look like a serious document surface.

### MemoryCandidateCard

Show what Tilo wants to remember:

```text
User prefers conservative contract review with explicit actionable revision suggestions.
```

Actions:

- Confirm memory
- Reject
- Edit placeholder

### ActionQueue

Show next actions:

- Generate revision draft
- Export review report
- Draft negotiation email
- Save this review style as a skill candidate

---

## 7. Interaction Rules

Important interactions must not be local-only.

When a user clicks approve, reject, confirm memory, or select an action:

1. Show loading state.
2. Call backend API if available.
3. Persist UIInteractionEvent / Confirmation / Memory state if supported.
4. Update UI status.
5. Add visible observation or trace update if possible.

If backend API is missing:

- mark as coming soon, or
- implement minimal safe endpoint, or
- use clearly labeled demo mode

Do not silently fake durable state.

---

## 8. Component Style Rules

### Cards

- use white surfaces
- subtle slate border
- 16-20px radius
- soft shadow
- clear header
- compact body

### Badges

Use clear semantic colors:

- high risk: red
- medium risk: amber
- low risk: blue/green
- completed: green
- pending: amber
- active: indigo/violet

### Buttons

- primary action: indigo/violet/blue
- destructive/reject: neutral or red depending context
- secondary: white with border
- disabled/future: muted

### Typography

- strong title hierarchy
- compact body text
- avoid long paragraphs inside cards
- use labels and metadata

---

## 9. Workflow Surface Rules

The center surface should feel like a workflow.

Use:

- stepper or timeline
- active node highlight
- completed state
- pending confirmation state
- muted future state
- components grouped under the relevant workflow node

Do not place random cards with no order.

---


## 11. Open-source Demo Standard

The UI should be good enough for:

- README screenshots
- GitHub social preview
- short launch video
- Product Hunt / Hacker News style sharing
- developer onboarding

A first-time visitor should understand in 10 seconds:

```text
Tilo is not a chatbot.
Tilo generates interactive SaaS workflows.
User interactions become observations.
Confirmed learning becomes memory.
```
